User Manual

Paper Blog is an open-source, full-stack blog platform built with FastAPI (backend) and Next.js 16 (frontend). Its defining feature is live theme switching: an admin pastes any external CSS URL and the site's appearance changes instantly for all visitors — no redeploy required.

This manual covers everything from first run to production deployment.

ℹ️
API documentation The backend exposes interactive API docs at http://localhost:8000/docs (Swagger UI) and http://localhost:8000/redoc (ReDoc) when running locally.

Prerequisites

You need two tools installed before running the project: uv for the Python backend and Node.js ≥ 20.9.0 for the frontend.

Backend — uv

The backend uses uv to manage the Python environment. Install it with:

bash
curl -LsSf https://astral.sh/uv/install.sh | sh

Verify with uv --version. Python 3.12+ is required (managed automatically by uv).

Frontend — Node.js ≥ 20.9.0

Next.js 16 requires Node.js 20.9.0 or higher. Use nvm to manage versions — the system Node from apt is often v18 which won't work.

bash
# Install nvm
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.0/install.sh | bash
source ~/.bashrc

# Install and use Node 24 LTS
nvm install 24
nvm use 24
⚠️
nvm in non-interactive shells nvm is sourced via your shell profile (.bashrc / .zshrc). It is not available in non-interactive shells unless you explicitly run source ~/.nvm/nvm.sh first.

Quick Start

Clone the repository, then start the backend and frontend in two terminal tabs.

Start the backend

The uv project root is backend/, so you must run from that directory.

bash
cd backend
uv run python main.py

On first run, backend/test.db (SQLite) is created automatically. The server runs on http://localhost:8000. To reset the database, delete test.db and restart.

Start the frontend

bash
source ~/.nvm/nvm.sh
nvm use 24
cd frontend
npm install   # first time only
npm run dev

The frontend runs on http://localhost:3000. Both services must be running simultaneously.

Reading Articles

No account is needed to read. The home page (/) shows all published articles ordered by date with title, author, and publication date. Click any article to open the full view.

Each article page includes:

  • A table of contents (hover the tab on the left to expand)
  • A reading progress bar at the top of the page
  • Like and share buttons in the right rail
  • A comments section at the bottom (reading login-free; posting requires an account)

Searching Articles

Use the search bar in the top navigation or go to /search-article. Search is partial and case-insensitive — typing fast will match articles titled "FastAPI Tips". Only published articles appear in results.

Registration & Login

Go to /register and fill in your email and password. After submitting you'll see a "Pending approval" message — your account is inactive until an admin activates it.

Once approved, log in at /login. The JWT token is stored in localStorage and expires after 1 hour. Log out via the navigation bar.

💡
Password reset Use the "Forgot password?" link on the login page. In development the reset token is returned in the API response (no email sent). Hook up SMTP to send real reset emails.

Getting Approved

After registering, your account has the guest role and is inactive. An admin must activate your account from the Admin → Users panel before you can log in and write. You'll also need the author role (or admin) to access the dashboard and create articles.

Register at /register

Submit your email and password. You'll see a pending message — you cannot log in yet.

Admin activates your account

An admin visits Admin → Users, finds your email, and clicks "Activate". Your role is updated to author.

Log in and start writing

Go to /login. You now have access to /dashboard and /create-article.

Writing Articles

Navigate to /create-article (or click "Write" in the navigation). The editor has two parts: a plain text title field at the top and the Editor.js block editor below.

Editor.js Blocks

Click the + icon to the left of any empty block, or type / to open the block menu. Available block types:

Draft vs Published

The Published toggle in the editor toolbar controls visibility. Drafts are only visible to the article's author and admins. Toggle it on to make an article public, off to revert to draft.

ℹ️
Only published = true articles appear on the public home page and in search results.

Managing Articles

Your dashboard at /dashboard lists all your articles with their publish status, like count, and comment count. From here you can:

  • Click the title to open the article
  • Click Edit to open it in the editor
  • Click Delete to permanently remove it

You can only manage your own articles. Admins can manage all articles from the admin panel.

Your Profile

Visit /profile to manage your personal information. The profile is divided into sections — click any row to open its edit modal:

  • Basic info — first name, last name, headline, bio
  • Work experience — job title, company, dates, description
  • Qualifications — certifications and credentials
  • Education — schools by grade level (Primary → University)

Admin Panel

The admin panel is at /admin and is only accessible to users with is_superuser = true. From the dashboard you can see a summary of total users, articles, and comments, then navigate to each management section.

⚠️
First admin The first user to register does not automatically become admin. Register an account through the UI first, then promote it to admin directly in the database — see the First Admin section in Deployment.

User Management

Navigate to Admin → Users. You'll see a table of all registered users with email, role, and active status.

  • Activate — enables the user to log in; sets their role to author
  • Deactivate — blocks the user from logging in without deleting their data

Role values: guest (default after registration), author (can write), admin (full access).

Article Management

Navigate to Admin → Articles to see all articles across all authors. Admins can edit or delete any article regardless of ownership. The table supports sorting and pagination.

Theme Management

This is the platform's signature feature. Navigate to Admin → Themes.

Add a theme

Enter a name and paste a publicly accessible CSS URL. Click Add Theme. The theme is saved but not yet active.

Activate a theme

Click Activate next to any saved theme. This deactivates all other themes and marks this one active. The change takes effect immediately — no redeploy needed.

Verify the change

Open the blog in a new tab. The new CSS is injected server-side on every page load via a <link> tag, so every visitor sees the updated look immediately.

💡
How it works technically The Next.js (blog)/layout.tsx server component calls GET /api/v1/theme/active on every request and injects <link rel="stylesheet" href="{url}" /> into <head>. No CDN or cache layer is involved — the URL is fetched fresh each time.

Site Settings

Navigate to Admin → Settings. Available options:

SettingDescription
Site NameAppears in the navigation wordmark and page titles
Logo URLURL of an image to show instead of the text wordmark (optional)
Allow RegistrationToggle off to hide the "Get started" / Register links for new users

Settings are fetched via the public GET /api/v1/settings endpoint — changes take effect immediately.

Comment Moderation

Navigate to Admin → Comments to view all comments across the platform. You can search by author or article and delete any comment. The comment list is paginated and sorted by newest first.

Deployment

Two deployment paths are supported: Docker Compose for local self-hosting, and Render Blueprint (render.yaml) for one-click cloud deployment.

Docker Compose — Local / Self-hosted

The docker-compose.yml at the repo root runs three services: PostgreSQL 16, the FastAPI backend, and the Next.js frontend. This is the recommended path for local development and self-hosted servers.

bash
docker compose up --build

Render — Blueprint Deployment

The repo includes a render.yaml Blueprint that provisions three separate Render services in one click: a managed PostgreSQL database, the FastAPI backend (Docker), and the Next.js frontend (native Node.js build).

Deploy via Blueprint

In the Render dashboard click New + → Blueprint, connect your GitHub repo, then click Apply. Render provisions all three services automatically. The first build takes 3–6 minutes.

Note your service URLs

Render may append a random suffix to service names if the plain name is already taken (e.g. blog-backend-2xzu.onrender.com). Note the actual URLs from each service's page before continuing.

Wire the URLs (Step 2 in render-deploy.md)

See Wire URLs below.

Create the first admin

See First Admin below.

ℹ️
The full step-by-step guide including post-deploy checklist is in render-deploy.md in the repo root.

Wire the backend ↔ frontend URLs

After the first deploy, each service needs to know the other's URL. Two env vars must be set manually because the URLs are only known after the services are created.

FRONTEND_URL — tells the backend which origin to allow for CORS

  1. Copy the blog-frontend service URL (e.g. https://blog-frontend-xxxx.onrender.com)
  2. Go to blog-backend → Environment tab → add FRONTEND_URL = https://blog-frontend-xxxx.onrender.com
  3. Save — the backend redeploys automatically

NEXT_PUBLIC_API_URL — baked into the Next.js client bundle at build time

  1. Copy the blog-backend service URL (e.g. https://blog-backend-xxxx.onrender.com)
  2. Go to blog-frontend → Environment tab → update NEXT_PUBLIC_API_URL = https://blog-backend-xxxx.onrender.com
  3. Save — the frontend rebuilds with the correct URL baked into the bundle
⚠️
NEXT_PUBLIC_API_URL is a build-time variable. Saving it triggers a full frontend rebuild. API calls from the browser will fail until the rebuild completes and the service is live again.

First Admin User

There is no admin account out of the box. The Render Shell (direct container access) is a paid feature, so the easiest approach is to use a PostgreSQL GUI tool such as TablePlus (free tier is sufficient).

Register an account

Go to /register on your deployed frontend and create the account you want to promote. This ensures the password is properly hashed by the application.

Get the database connection string

In the Render dashboard go to blog-db → Info tab → copy the External Connection String (format: postgresql://blog_user:PASSWORD@HOST/blog_db).

Connect with TablePlus

Open TablePlus → New Connection → PostgreSQL. Paste the connection string into the URL field. Enable SSL (Render requires it). Click Connect.

Run the promotion query

Open a new query tab (Cmd/Ctrl+T) and run:

sql
UPDATE users
SET is_superuser = true,
    is_active    = true,
    role         = 'admin'
WHERE email = 'your@email.com';

Click Commit (Cmd/Ctrl+S) — TablePlus does not auto-commit.

Log in

Go to /login on your deployed frontend. You now have access to /admin.

Troubleshooting

ModuleNotFoundError: No module named 'uvicorn'

Cause: You ran uv run from the repo root instead of backend/.
Fix: cd backend first, then uv run python main.py.

Node.js version error on frontend start

Cause: System Node is v18 (below the required ≥ 20.9.0).
Fix:

bash
source ~/.nvm/nvm.sh && nvm use 24

CORS errors in the browser

Cause: The backend's FRONTEND_URL doesn't match the origin the browser is sending.
Fix: Set FRONTEND_URL to your exact frontend origin (e.g. https://blog-frontend-xxxx.onrender.com) in the backend environment on Render.

Login succeeds but I'm redirected back to login

Cause: The JWT token isn't being saved to localStorage — this can happen if CORS credentials are misconfigured or the token response format changed.
Fix: Open browser DevTools → Network → check the /auth/jwt/login response. Ensure access_token is present in the JSON.

Theme change not visible to visitors

Cause: The CSS URL is not publicly accessible or returned a non-200 status.
Fix: Open the URL directly in a browser and confirm it returns valid CSS. Also ensure the URL is HTTPS in production.

Render — backend migration fails on first deploy

Cause: The DATABASE_URL Render injects uses the postgres:// scheme, but SQLAlchemy's async engine requires postgresql+asyncpg://. The app normalises this automatically.
If you see an SSL error ("certificate verify failed: self-signed certificate"), ensure you are on the latest commit — earlier versions passed ssl=True (boolean) which triggers strict verification. The fix uses an explicit ssl.SSLContext with CERT_NONE.

Render — frontend returns 502 Bad Gateway

Cause: The Next.js standalone server is binding to localhost instead of 0.0.0.0, so Render's reverse proxy can't reach it.
Fix: Ensure HOSTNAME=0.0.0.0 is set in the blog-frontend environment — this is already included in render.yaml. If you added the service manually, add this variable in the Render dashboard.

Render — frontend build fails with "Cannot find module 'typescript'"

Cause: npm ci with NODE_ENV=production skips devDependencies. TypeScript is a devDependency required to transpile next.config.ts.
Fix: The build command in render.yaml uses npm ci --include=dev to force devDependency installation. If you see this error, confirm your render.yaml build command includes the --include=dev flag.

Render — API calls return 404 after deploying

Cause: NEXT_PUBLIC_API_URL is baked into the Next.js bundle at build time. If it was set after the initial build or points to the wrong backend URL, all client-side API calls will fail silently.
Fix: Update NEXT_PUBLIC_API_URL in the frontend environment to match your actual backend URL (including any Render suffix), then save to trigger a rebuild.

Render — free tier "cold start" delay

Cause: Render free services spin down after 15 minutes of inactivity. The first request after spin-down takes 30–60 seconds.
Fix: This is expected on the free tier. Upgrade to a paid plan to keep the service always-on, or use an external uptime monitor to keep it warm.
