User Manual

Paper Blog is an open-source blog platform built on FastAPI and Next.js 16. It ships with a rich block editor (Editor.js), role-based authentication, live theme switching, and a built-in admin panel — all packaged so a single operator can run a small publication without writing code.

This manual is for people using a deployed instance. It is split into four parts: Readers (anyone visiting the site), Authors (people writing posts), Admins (the people approving authors and configuring the site), and Operators (the person who deployed the thing). Pick the section that matches your hat.

Quick Start

If you have just landed on a deployed Paper Blog, here is the shortest path to participating.

  1. Open the site URL — the home page lists the most recent published articles.
  2. Click any title to read the full article. No account needed.
  3. Use the search bar (or visit /search-article) to find articles by title.
  4. To leave a like or comment, register at /register with your email and a password.
  5. After registering you will see a "pending" message — your account is inactive until an admin approves it.
  6. Once approved, sign in at /login. Your token is stored in localStorage and lasts one hour.
  7. If the admin also gave you the author role, the navigation will expose Write and Dashboard.
  8. Admins land on /admin after sign-in.

For Readers

You can read every published article without an account. Likes, comments, and bookmarks all require sign-in, but nothing about reading does.

Reading & Article View

The home page (/) shows only published articles, ordered newest first, paginated 20 per page. Drafts never appear here, regardless of who wrote them.

Each article page is a three-column reading view:

  • Title, byline (author name + avatar), publication date, computed read time, and word count.
  • A table of contents rail on the left that auto-builds from the article's headings.
  • A reading-progress bar at the very top of the page that fills as you scroll.
  • A right-side rail with Bookmark, Share, and the read-time badge.
  • A Reactions section (likes) and a threaded Comments section at the bottom.

Articles also emit JSON-LD structured data, so they show up cleanly in search engines and link previews.

i
Anonymous reading is fine. You can read, browse, and search without logging in. Liking and commenting are the first things that ask for an account.

Search

The search page lives at /search-article. It is a title-only, case-insensitive substring match — typing fast matches an article called "FastAPI Tips" but not one whose title is "Speedy Servers".

Under the hood it calls GET /api/v1/search-articles?title=…. Search returns up to a single page of matching published articles in one shot — there is no pagination on the search endpoint, so very broad queries on a large site will be capped server-side.

Register & Sign In

Registration is at /register. You only need an email address and a password. There are no email-verification steps to clear, but there is an approval gate — see below.

New accounts are inactive by default. The backend's UserManager.on_after_register hook flips is_active=false the moment your row is created. Until an admin activates the account, the login page rejects you with an "inactive user" / 401 error. This is by design — it stops drive-by signups from posting comments.

If the admin has switched off allow_registration in Site Settings, the registration link disappears and the form is hidden. Reach out to the operator if you need an account on a closed instance.

+
Forgot password. Use the link on the login page. In a default install, the reset token is returned in the API response (no email is sent) — the operator wires up SMTP themselves.

For Authors

Authors are users who can create, edit, and publish articles. The path from "I just registered" to "I can publish" goes through one human gate (an admin) and two server-side checks (active flag + author role).

!
Activation alone is not enough. Clicking "Activate" in the admin user list only sets is_active=true. It does not change your role from guest to author. If you can log in but the dashboard 403s, ask your admin to bump your role — see the User Management section below for the exact path.

Writing with Editor.js

Click Write in the navigation, or go directly to /create-article. The screen has a plain text title field at the top and the Editor.js block editor below it. Click the + icon next to any block, or type / on an empty line to open the block menu.

Drafts vs Published

The Published checkbox in the editor toolbar is the visibility switch. Internally it is a string column — "true" or "false" — not a boolean, so the backend does string comparisons everywhere. From your side, it is just a checkbox.

  • Draft (unchecked): visible only to you and to admins. Will not appear on /, in search, in the sitemap, or in RSS.
  • Published (checked): visible to everyone, included in the home feed, search results, and sitemap.xml.

You can flip the toggle as many times as you like — un-publishing pulls the article straight back to draft.

Editing & Deleting

Your Dashboard at /dashboard lists every article you own — drafts and published, with their like and comment counts. From there:

  • Click the title to open the live article view.
  • Click Edit to reopen the editor at /update-article/{id}.
  • Click Delete to remove the article permanently. There is no soft-delete or undo.

Ownership is checked server-side. If you try to PUT or DELETE someone else's article you will get a 403 — not even a clever cURL gets around this. Admins can edit and delete any article via /admin/articles.

Your Profile

Visit /profile. The page is split into four cards, each with its own edit modal.

For Admins

Admins (users with is_superuser=true) get an extra panel at /admin. Everything in this section assumes you have already been promoted to admin — see Bootstrapping the First Admin if not.

Admin Dashboard

The dashboard lives at /admin and is the landing page for the panel. It surfaces a handful of KPIs powered by GET /api/v1/admin/stats:

  • Articles total, plus a split into published and draft.
  • Users total, plus an active count (i.e. how many have been approved).

From here, the side navigation routes you to Articles, Comments, Users, Theme, and Site Settings.

User Management

Open /admin/users. You will see a paginated table (20 per page) with email, status (Active / Pending), role badge, and an action button. The search box filters by email substring and is case-insensitive; GET /api/v1/admin/users?q=… does the work.

Two actions are exposed in the UI:

  • Activate — calls PATCH /api/v1/admin/users/{id}/activate. Sets is_active=true. The user can now log in.
  • Deactivate — calls PATCH /api/v1/admin/users/{id}/deactivate. Sets is_active=false. The user's data is kept but they can no longer authenticate.
!
Role assignment is not in the UI. The Users page does not currently expose a way to change a user's role from guest to author. The activate flow does not change the role either. To promote a reader to an author, run SQL directly against the database:
sql
UPDATE users SET role = 'author' WHERE email = 'someone@example.com';

Use psql, TablePlus, DBeaver, or the connection string from your hosting provider's database panel. Pair this with Activate in the UI, and the user becomes a fully working author.

Theme Management

The platform's signature feature lives at /admin/theme. A theme is a row in the theme_configs table with a name and an external CSS url. Exactly one theme is active at a time.

  1. Click Add Theme, enter a name and a publicly reachable CSS URL, save. The theme is registered but not active.
  2. Click Activate next to any theme. This calls PUT /api/v1/admin/themes/{id}/activate, which deactivates every other theme and flips this one on.
  3. Open the public site in a new tab. The blog layout server component injects <link rel="stylesheet" href="{url}"> into <head> on every page render.

The active-theme fetch is server-cached with revalidate: 60, so visitors will see the new theme within roughly a minute even without a hard refresh.

+
Where to host the CSS. Any HTTPS URL that returns a valid CSS file works. Common choices: a GitHub repo's raw.githubusercontent.com URL, a jsDelivr CDN URL pointing at an npm-published theme, your own static host, or a CSS gist on jsDelivr. If the URL is HTTP-only or returns non-200, the link will load but the site will fall back to base styles.

Site Settings

The settings page edits the singleton SiteConfig row. The same row is exposed publicly at GET /api/v1/settings (the frontend uses it for branding and metadata) and edited via PUT /api/v1/admin/settings.

Content Moderation

Two pages, both under /admin:

  • /admin/articles — every article on the site, regardless of author or publish status. Search, paginate, edit, delete. Any deletion here is permanent.
  • /admin/comments — every comment on every article. Search by comment body or article title; delete with one click. Sorted newest first.

There is currently no spam filter or rate-limit. If a deployed instance starts attracting drive-by comments, the practical mitigation is to flip allow_registration off and only invite the people you trust.

Deployment for Operators

Paper Blog deploys as a single Docker image: nginx + supervisord fronting the FastAPI backend and the Next.js frontend in one container. The same image runs in docker compose locally and on Render. Pick whichever path matches where you want the site to live.

Deploy on Render (Web Service)

This is the manual route — useful if you want to set the service up by hand instead of using the Blueprint. The repo's render.yaml defines the same thing declaratively if you'd rather click New + → Blueprint.

  1. Create the database first. Render dashboard → New + → PostgreSQL. Name blog-db, region Oregon, plan Free. Wait for the Available status.
  2. Copy the Internal Database URL. From the database's Info tab, grab the Internal Connection String. It looks like postgresql://blog_user:…@dpg-xxxx-a/blog_db.
  3. Create the Web Service. New + → Web Service → Docker. Connect your GitHub repo. Branch: master. Root directory: leave blank. Dockerfile path: ./Dockerfile. Build context: . (a single dot — the repo root).
  4. Add environment variables.
    • DATABASE_URL — paste the Internal URL from step 2.
    • SECRET — generate one with openssl rand -hex 32. Do not reuse the dev default.
    • FIRST_ADMIN_EMAIL — optional, but the easiest way to bootstrap an admin (see below).
  5. Add a persistent disk. Disks tab → Add Disk. Name uploads, mount path /app/backend/uploads, size 1 GB. Without this, every redeploy wipes uploaded logos.
  6. Health Check Path. Set it to /healthz. Nginx serves this directly without touching the backend.
  7. Create Web Service. Render builds the Dockerfile (5–10 minutes the first time) and starts the container.
!
Same project for the Internal URL. The web service and the database must live in the same Render project (workspace) for the Internal URL hostname (dpg-xxxx-a) to resolve. If they are split across projects, switch DATABASE_URL to the External connection string instead — it is slower but works from anywhere.

Self-host with Docker

The docker-compose.yml at the repo root brings up Postgres 16 and the consolidated app container. This mirrors the Render deployment exactly, but locally.

bash
git clone https://github.com/<your-fork>/blog.git
cd blog
docker compose up -d

Once both containers are healthy, browse to http://localhost:8080. Nginx is the only public listener; the backend (8000) and frontend (3000) are not exposed.

To bootstrap the first admin via env var, set FIRST_ADMIN_EMAIL when bringing the stack up:

bash
FIRST_ADMIN_EMAIL=you@example.com docker compose up -d

Register that email through the UI first so the password is hashed correctly, then restart the app container — the lifespan handler will promote the row.

Bootstrapping the First Admin

There is no admin out of the box. You have two paths to create the first one.

Path A — FIRST_ADMIN_EMAIL auto-promote

  1. Go to /register on the live site and create an account with the email you want to be admin. (You will see "pending approval" — that is fine.)
  2. Set FIRST_ADMIN_EMAIL on the service to that exact email.
  3. Trigger a redeploy / restart. The lifespan handler in backend/app/app.py finds the row and sets is_superuser=true, is_active=true, role='admin'.
  4. Sign in at /login and visit /admin.

Path B — direct SQL

Useful when you can't redeploy, or you want to avoid promoting on every boot.

sql
UPDATE users
SET is_superuser = TRUE,
    is_active    = TRUE,
    role         = 'admin'
WHERE email = 'you@example.com';

Run this against the database via Render's database shell, psql with the External URL, or any Postgres GUI (TablePlus, DBeaver). TablePlus does not auto-commit — press Cmd/Ctrl+S after running.

Persistent Uploads & Backups

Uploaded logos (and any future avatar uploads) are written to /app/backend/uploads/ inside the container and served back at /uploads/{filename} by FastAPI's StaticFiles mount.

  • On Render, attach a 1 GB Disk mounted at /app/backend/uploads (the render.yaml Blueprint does this for you). Without the disk, every redeploy wipes uploads.
  • In docker-compose, the uploads named volume covers the same path. docker compose down keeps the volume; docker compose down -v destroys it.

Database backups are your responsibility. Nothing in the app does this for you. Render's managed Postgres has a backup feature on paid plans (Settings → Backups). For self-hosted Postgres, run pg_dump on a cron:

bash
pg_dump "$DATABASE_URL" | gzip > "blog-$(date +%F).sql.gz"
!
Free Render Postgres expires after 90 days. Export your data or upgrade to the Starter plan before then, otherwise the database is deleted.

Troubleshooting Production

1. Alembic fails with socket.gaierror: Name or service not known

What it means. The container started, called alembic upgrade head, and could not resolve the hostname inside DATABASE_URL to an IP.

Fix. If you are on Render, the most common cause is that the web service and the database are in different projects, so the Internal URL (dpg-xxxx-a) does not resolve. Either move both into the same project, or switch DATABASE_URL to the External connection string. The container intentionally crashes loud here rather than booting against a missing schema — that is a feature, not a bug.

2. nginx: "daemon" directive is duplicate

What it means. Something is starting nginx with both daemon off baked into the config and a -g 'daemon off;' CLI override. The shipped nginx.conf.template has daemon off; in the http block; supervisord.conf launches nginx without an extra -g.

Fix. Make sure deploy/supervisord.conf has not been edited to add -g 'daemon off;' back into the [program:nginx] command. The correct command is /usr/sbin/nginx -c /etc/nginx/nginx.conf.

3. Container exits before serving traffic

What it means. Either the nginx config validation (nginx -t) failed, or alembic crashed.

Fix. Read the logs from the top. Lines tagged [start.sh] show progress through nginx-render → uploads-dir → alembic-migrations → supervisord-handoff. The first non-[start.sh] error after one of those lines is the real failure. Migration failures are intentional — the container will not boot against a broken schema.

4. Free-tier sleep on Render

What it means. Render's free web services spin down after 15 minutes of inactivity. The first request after a sleep takes 30–60 seconds while the container cold-boots.

Fix. Expected behaviour. Upgrade to Starter to keep the service always-on, or run a periodic external uptime monitor that pokes /healthz every 10 minutes. Note that this also keeps the free Postgres from idling.

5. CORS errors in the browser

What it means. The backend's FRONTEND_URL does not match the origin the browser is calling from.

Fix. In single-image mode you usually do not need FRONTEND_URL at all because the API and the browser share an origin. If you do set it, set it to the exact public URL of your service, scheme included, no trailing slash.

FAQ

Why was my registration not approved automatically?

By design. UserManager.on_after_register sets is_active=false on every new account so admins can vet who joins. Ask the admin to activate you, or — if you are the operator — see the bootstrap section.

I was activated but still can't write articles.

You have is_active=true but probably still have role='guest'. The "Activate" button does not change role. Get an admin to run UPDATE users SET role='author' WHERE email='you@example.com'; against the database.

How do I change a user's role from the UI?

You can't, today. The Users panel only exposes Activate/Deactivate. Role assignment is a direct database edit. The brief on this UX has been raised; for now, SQL is the answer.

Can I run this on SQLite in production?

It works, but it's not recommended. SQLite cannot handle concurrent writers as well as Postgres, and the SQLite file lives inside the container by default — a stateless redeploy will erase it unless you mount the file onto a persistent disk. Use Postgres for any deployment you care about.

Where do uploaded images in my articles go?

Nowhere — the platform does not host article images. The Editor.js Image block expects a URL pointing at an already-hosted image (Imgur, S3, GitHub raw, etc.). The only built-in upload is the admin Logo upload, which goes to /app/backend/uploads/.

How do I rotate SECRET?

Change the env var and redeploy. Every existing JWT becomes invalid immediately — every signed-in user is logged out and has to re-authenticate. Plan to do this off-hours.

How long does a session last?

One hour. The JWT carries a 3600-second lifetime, and the frontend stores it in localStorage as access_token. There is no refresh-token flow today; users sign in again after expiry.

How do I disable new registrations entirely?

Admin → Site Settings, untick Allow Registration, save. The register links disappear from the navigation and the form is hidden. The endpoint is still reachable directly, but the UI no longer advertises it.

The theme I activated isn't showing for visitors.

Either the CSS URL returned a non-200, or the URL is not HTTPS while the site is HTTPS (mixed content gets blocked). Open the URL in a browser and confirm it returns valid CSS, then give the server-side cache up to 60 seconds to revalidate.

Where are the API docs?

/docs on your deployed origin (Swagger UI), or /openapi.json for the raw schema. Nginx routes both straight through to FastAPI in production, so they work everywhere the app does.

Project Overview

Paper is an open-source full-stack blog platform with role-based authoring, dynamic themes, and a single-container deploy story. The backend is a layered FastAPI app; the frontend is a Next.js 16 App Router project. The two ship together inside a single Docker image (nginx + supervisord) for one-step Render / VPS deploys, or you can run them split via docker compose against Postgres for local parity. Released under the MIT License (LICENSE at the repo root).

At a glance: a thin api/v1/* layer hands work to services/*, which own transactions and ownership checks against SQLAlchemy models defined in app/database/db.py. The frontend talks to the backend via fetch wrappers in app/_lib/api_callout.js and stores a JWT in localStorage. See Tech Stack and Architecture for the full picture.

Architecture

Three boundaries to keep in mind: (1) FastAPI request handler (api/v1/*) — thin, no business logic; (2) service layer (services/*) — owns transactions, validation, ownership checks; (3) ORM (database/db.py) — async SQLAlchemy. The frontend is pure App Router with three route groups (auth) / (blog) / (admin), a server-rendered theme + metadata pipeline, and a localStorage JWT.

Backend Layering

layout
HTTP request
    │
    ▼
api/v1/*.py            ← thin route handlers
    │                    Depends(...) for auth / session
    ▼
services/*.py          ← business logic, transactions, ownership
    │                    raises HTTPException on bad input
    ▼
database/db.py         ← async SQLAlchemy models + session factory

Route files: articles.py (CRUD, likes, comments, sitemap), admin.py (users, themes, stats, settings, comments, logo upload), author.py (author dashboard), profile.py (profile, experience, qualifications, school).

Service files mirror the routers: services/articles.py, services/admin.py, services/theme.py, services/site_settings.py, services/profile.py. Services take an AsyncSession as the first arg and return dicts or ORM instances. Routes never call session.commit() directly.

ℹ️
Pydantic schemas Live in app/models/ (articles.py, theme.py, users.py). Not SQLAlchemy models — those live in app/database/db.py. The naming overlap is unfortunate; just remember app/models/ = Pydantic, app/database/ = SQLAlchemy.

Frontend Layering

Next.js App Router with three route groups, each with its own layout and globals.css:

  • app/(auth)/ — login, register, forgot-password
  • app/(blog)/ — public feed, article view, search, plus authenticated create-article, update-article/[articleId], dashboard, profile
  • app/(admin)/ — admin panel (articles, comments, users, theme, site settings) wrapped by AdminGuard.js

Shared bits:

  • app/_lib/api_callout.js — fetch wrapper for every backend endpoint (one function per route)
  • app/_lib/theme.ts — server-side fetchers for active theme + public site config, cached with next: { revalidate: 60 }
  • app/_lib/utility.js — JWT decoding (getTokenFromLocalStorage) and the date formatter the backend expects
  • app/context/AuthContext.tsx — auth state. On mount, reads access_token from localStorage and verifies it against GET /users/me; clears the token on failure

NEXT_PUBLIC_API_URL defaults to "" in api_callout.js so production single-image deploys use relative URLs (same-origin via nginx). Server-side fetchers in theme.ts and (blog)/layout.tsx default to http://localhost:8000 for local dev.

Auth & JWT Flow

flow
1. POST /auth/register                  → user created
2. UserManager.on_after_register        → is_active = False
3. PATCH /api/v1/admin/users/{id}/activate (admin)
4. POST /auth/jwt/login                  → { access_token, token_type }
5. Frontend stores access_token in localStorage
6. AuthContext.tsx GET /users/me on every mount
   ─ 200 → context.user populated
   ─ 401 → token cleared, redirect to /login

JWT lifetime is 3600 seconds (1 hour), signed with the SECRET env var (default dev-secret-change-me — must override in prod). See backend/app/core/users.py.

Three guard layers, used as FastAPI Depends(...):

  • current_active_user — any logged-in, active user
  • current_author_or_admin — 403 unless is_superuser or role in ("author", "admin")
  • current_superuser — admin-only routes (/api/v1/admin/*)

Ownership is then re-checked inside services — for example services/articles.py:

python
if not user.is_superuser and post.owner_id != user.id:
    raise HTTPException(status_code=403, detail="Not your post")

Theme System

A "theme" is just a CSS file at a public URL. Admins register a row in the ThemeConfig table; one row at a time may be active. The flow:

  1. POST /api/v1/admin/themes with { name, url } — creates a row
  2. PUT /api/v1/admin/themes/{id}/activate — flips is_active, clears it on every other row
  3. Frontend app/_lib/theme.ts server-fetches /api/v1/theme/active with next: { revalidate: 60 }
  4. (blog)/layout.tsx renders <link rel="stylesheet" href={themeUrl} /> in the head

A change reflects within ~60 seconds (cache TTL). Commit f43db54 added a --font-body CSS variable (default var(--font-serif)) to each route group's globals.css so themes can override the body typography without touching every .article-body rule individually.

Site Settings & Singletons

SiteConfig is a singleton row (id = 1). Fields: site_name, site_description, site_url, logo_url, allow_registration, og_title, og_description, og_image_url.

Two endpoints:

  • GET /api/v1/settings — public, returns the whole row, used by the frontend to generate <title>, OpenGraph tags, etc.
  • PUT /api/v1/admin/settings — superuser-only, partial update (any field may be omitted)

Logo uploads go through POST /api/v1/admin/upload-logo (multipart) which writes to backend/uploads/logo_{uuid}.{ext} and returns { url: "/uploads/..." }. The backend mounts that directory at /uploads/ via StaticFiles; nginx forwards /uploads/ to the backend in single-image deploys.

Single-Image Deploy Topology

One container, three processes managed by supervisord. Only nginx is publicly bound; uvicorn and Next.js listen on loopback.

topology
           ┌──────────────────────────────────────────────┐
           │           Container (single image)            │
           │                                                │
public  ──▶│  nginx :$PORT (default 8080)                  │
($PORT)    │     │                                          │
           │     ├── /api/v1/   ┐                           │
           │     ├── /auth/     │                           │
           │     ├── /users/    ├──▶ 127.0.0.1:8000        │
           │     ├── /uploads/  │      uvicorn (FastAPI)   │
           │     ├── /docs      │                           │
           │     └── /openapi.json                          │
           │                                                │
           │     └── /  (catch-all) ──▶ 127.0.0.1:3000     │
           │                              Next.js standalone│
           └──────────────────────────────────────────────┘

Boot sequence (deploy/start.sh):

  1. envsubst '$PORT' renders nginx.conf.template → /etc/nginx/nginx.conf
  2. nginx -t validates the rendered config (loud failure)
  3. mkdir -p /app/backend/uploads
  4. uv run --no-sync alembic upgrade head (skipped if DATABASE_URL unset)
  5. exec /usr/bin/supervisord -c /deploy/supervisord.conf

Local Setup

Two terminal flows: native (fastest hot-reload, recommended for dev) or docker compose (one command, mirrors production).

Prerequisites

  • Python 3.12+ — managed automatically by uv
  • Node.js 20.9+ — Next.js 16 refuses lower; install via nvm
  • uv — curl -LsSf https://astral.sh/uv/install.sh | sh
  • Docker (optional) — only needed for the compose flow or for running Postgres locally
  • PostgreSQL (optional) — SQLite is the default; Postgres is closer to production

Backend (Python + uv)

bash
cd backend
uv run uvicorn app.app:app --reload --host 0.0.0.0 --port 8000

Or just uv run python main.py. Must be run from backend/ — that's where pyproject.toml lives, and uv walks up from the cwd to find it. Running from repo root fails with ModuleNotFoundError: No module named 'uvicorn'.

Defaults: DATABASE_URL=sqlite+aiosqlite:///./test.db (creates backend/test.db on first run), SECRET=dev-secret-change-me. backend/.env is auto-loaded via python-dotenv — drop overrides there.

ℹ️
Swagger UI at http://localhost:8000/docs, ReDoc at /redoc, raw OpenAPI at /openapi.json.

Frontend (Node + Next.js)

bash
source ~/.nvm/nvm.sh && nvm use 20  # or 24
cd frontend
npm install        # first time
npm run dev        # turbopack on :3000

The browser-side fetch wrapper (app/_lib/api_callout.js) defaults NEXT_PUBLIC_API_URL to "", producing relative URLs. In dev that means the browser tries http://localhost:3000/api/v1/... — which doesn't exist. Set NEXT_PUBLIC_API_URL=http://localhost:8000 in frontend/.env.local:

env
# frontend/.env.local
NEXT_PUBLIC_API_URL=http://localhost:8000
NEXT_PUBLIC_APP_URL=http://localhost:3000

Server-side fetchers in theme.ts and (blog)/layout.tsx already default to http://localhost:8000, so SSR works without that env var. CORS is handled by the FRONTEND_URL env var on the backend (defaults to http://localhost:3000).

Database (SQLite or Postgres)

SQLite (default, zero setup). Stored in backend/test.db. Reset by deleting the file.

env
DATABASE_URL=sqlite+aiosqlite:///./test.db

Postgres. Spin one up with Docker:

bash
docker run -d --name blog-pg \
  -e POSTGRES_USER=blog_user \
  -e POSTGRES_PASSWORD=blog_pass \
  -e POSTGRES_DB=blog_db \
  -p 5432:5432 \
  postgres:16
env
DATABASE_URL=postgresql+asyncpg://blog_user:blog_pass@localhost:5432/blog_db

The backend's URL normaliser (_async_db_url() in database/db.py) rewrites postgres:// and postgresql:// to postgresql+asyncpg:// automatically, so a Render-style postgres:// URL works as-is. It also strips ssl= and sslmode= query params (asyncpg mishandles them) and passes SSL via connect_args instead.

Docker Compose (full stack)

bash
docker compose up -d --build
# open http://localhost:8080

Spins up Postgres 16 plus the consolidated single-image app (nginx + uvicorn + Next.js) on port 8080. The app service depends on db with a healthcheck (pg_isready); migrations run automatically inside the container on boot.

No hot-reload Compose builds the production image — code changes require docker compose up --build. For active development, use the bare-metal flow above.

Gotchas — Must Read

  • Run backend from backend/, not the repo root. uv won't find pyproject.toml otherwise.
  • Post.published is a String ("true"/"false"), not a boolean. Services call str(post.published).lower() before saving and compare with Post.published == "true". Frontend handles both "true" and true.
  • ArticleBase.created_date is a string in "%Y-%m-%d %H:%M:%S" format. The service calls datetime.strptime with that exact format and 400s on mismatch. Use getCurrentFormattedDate() from frontend/app/_lib/utility.js.
  • UUIDs use fastapi_users_db_sqlalchemy.generics.GUID (native UUID on Postgres, CHAR(32) on SQLite). Don't replace with UUID(as_uuid=True) — you'll break SQLite.
  • New registrations are inactive by default. UserManager.on_after_register sets is_active=False; an admin must PATCH /api/v1/admin/users/{id}/activate before the user can log in.
  • reactStrictMode: false in frontend/next.config.ts is intentional — Editor.js double-init breaks under Strict Mode. Don't re-enable.
  • Postgres SSL params are stripped from the URL and passed via connect_args instead. asyncpg ignores sslmode= and mishandles ssl= in URLs. If you switch DB drivers, replicate this.
  • SECRET defaults to dev-secret-change-me — must be overridden in production.
  • NEXT_PUBLIC_API_URL is unset in the production single-image deploy — that's deliberate. The frontend bundle uses relative URLs which nginx routes to the backend on the same origin.

How To

Add a Backend Route

Walk-through: add GET /api/v1/articles/{id}/word-count.

  1. Add a service function in backend/app/services/articles.py:
python
async def get_word_count(session: AsyncSession, article_id: str) -> dict:
    post = await session.get(Post, article_id)
    if not post:
        raise HTTPException(status_code=404, detail="Article not found")
    # naive — strip Editor.js JSON and split on whitespace
    text = post.content or ""
    return {"id": str(post.id), "count": len(text.split())}
  1. Wire the route in backend/app/api/v1/articles.py:
python
from app.services.articles import get_word_count as svc_get_word_count

@router.get("/articles/{id}/word-count")
async def article_word_count(
    id: str,
    session: AsyncSession = Depends(get_async_session),
    user: Optional[User] = Depends(current_optional_user),
):
    return await svc_get_word_count(session, id)
  1. Add a fetch wrapper in frontend/app/_lib/api_callout.js:
javascript
export async function getWordCount(id) {
    const res = await fetch(`${API}/api/v1/articles/${id}/word-count`);
    const data = await res.json().catch(() => null);
    return res.ok ? { success: true, detail: data } : { success: false, detail: data };
}
  1. Use it from a component ("use client" page or effect):
javascript
import { getWordCount } from "@/app/_lib/api_callout";

const { detail } = await getWordCount(articleId);
console.log(detail.count);

The backend hot-reloads (--reload); the frontend hot-reloads via Turbopack. Verify in /docs first, then exercise from the UI.

Add a Frontend Page

Create frontend/app/(blog)/stats/page.js:

javascript
"use client";
import { useEffect, useState } from "react";
import { getArticles } from "@/app/_lib/api_callout";

export default function StatsPage() {
  const [state, setState] = useState({ loading: true, error: null, total: 0 });

  useEffect(() => {
    getArticles({ skip: 0, limit: 1 }).then(({ success, detail }) => {
      if (!success) return setState({ loading: false, error: "Failed", total: 0 });
      setState({ loading: false, error: null, total: detail.total });
    });
  }, []);

  if (state.loading) return <p>Loading…</p>;
  if (state.error)   return <p>{state.error}</p>;
  return <h1>{state.total} articles published</h1>;
}

It's now live at http://localhost:3000/stats. Add a link to it from frontend/app/(blog)/components/NavLinks.js (or wherever your nav lives) if you want it discoverable.

Write a Database Migration

bash
cd backend
# autogenerate based on model diff
uv run alembic revision --autogenerate -m "add tag column to posts"
# inspect the generated file under alembic/versions/
# apply
uv run alembic upgrade head
# roll back the most recent
uv run alembic downgrade -1

Test against both SQLite and Postgres before committing — autogenerate occasionally misses dialect-specific quirks (default values, JSON columns, indexed nullables). Run upgrade head from a fresh DB on each.

ℹ️
app/app.py also calls create_db_and_tables() on startup as a safety net for fresh SQLite installs. That runs metadata.create_all — it will not alter existing tables. For schema changes against an existing DB, you must write a real migration.

Build & Test a Theme

A theme is a CSS file at any reachable URL. It overrides variables and selectors defined in frontend/app/(blog)/globals.css (and the equivalents in (auth) / (admin)).

  1. Author a CSS file. Override CSS variables (--paper, --ink, --accent, etc.) and the new typography variable --font-body (commit f43db54).
  2. Host it where the browser can reach it: GitHub raw, jsDelivr (https://cdn.jsdelivr.net/gh/<you>/<repo>@main/theme.css), Cloudflare Pages, an S3 bucket, anywhere.
  3. Register it via the admin panel (/admin/theme) or the API:
bash
curl -X POST http://localhost:8000/api/v1/admin/themes \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"my-theme","url":"https://cdn.example.com/theme.css"}'
  1. Activate it (UI button or PUT /api/v1/admin/themes/{id}/activate).
  2. The browser sees the change after the next revalidation tick (~60s) — or a hard refresh.

Tip: serve the CSS locally via python -m http.server while iterating, then publish once it's stable.

Cloud Deployment for Forks

Two paths once you've forked the repo. Both build the consolidated single-image container.

1. Render (managed Web Service)

The full click-by-click is in the operator manual — see User Manual → Deploy on Render (#um-deploy-render). The only fork-specific step is pointing the Web Service at your fork's repo URL instead of the upstream. Everything else (envs, Postgres add-on, FIRST_ADMIN_EMAIL) is identical.

2. Self-hosted Docker / VPS

Any Linux VPS with Docker installed. Five commands:

bash
git clone https://github.com/<you>/blog.git
cd blog
# edit docker-compose.yml — at minimum override SECRET and FIRST_ADMIN_EMAIL
docker compose up -d --build
# verify
curl http://localhost:8080/healthz

Point your domain's A record at the VPS, then put a TLS-terminating reverse proxy in front. Caddy handles certs automatically:

caddyfile
blog.example.com {
    reverse_proxy localhost:8080
    encode gzip zstd
}

Set FRONTEND_URL=https://blog.example.com on the app service so CORS allows your real origin. Bind Postgres to a non-public network if you don't want it reachable from outside the host.

Submitting a Pull Request

  1. Fork the repo on GitHub.
  2. Branch off master: feature/add-tag-support, fix/sitemap-cache-header, docs/<…>. One concern per branch.
  3. Commit. Subject line ≤ 72 chars, imperative, lower-case verb. Examples: add tag column to posts, fix sitemap cache TTL. Body wraps at 72 cols and explains why, not what.
  4. Lint & migrate before pushing.
bash
cd frontend && npm run lint
cd ../backend && uv run alembic upgrade head    # against a fresh DB
  1. Push to your fork and open a PR against master.
  2. PR body must include:
    • What changed (bullet list)
    • Why — what problem does this solve, what's the alternative
    • Screenshots / GIFs for any UI change
    • Test plan — a checkbox list of manual steps to verify the change
    • Migration impact (yes/no — if yes, link the alembic file)

Keep PRs small and focused. A 60-line PR will be reviewed today; a 600-line one may sit. If a refactor and a feature are tangled, split them.
